## Observations

1. Signature parity is already pinned at 1.0 for all four working pairs — every
   stainless symbol that fern also emits has matching parameter signatures.
   The lever for tier-2 movement is **symbol-coverage** (weight 0.3) and
   **file-coverage** (weight 0.1), not signature.

2. Symbol gap is concentrated in three categories:
   - **Client class naming.** Stainless emits `Papr` / `Honcho` / `LumaAI` as
     the top-level client; fern emits `PaprApiClient` / `HonchoApiClient` etc.
     Renaming would break existing fern customers, so this would have to go
     behind an opt-in config flag with a generator migration to flip the
     default in the next major.
   - **Error subclasses.** Stainless emits a class per HTTP status family
     (`BadRequestError`, `AuthenticationError`, `PermissionDeniedError`,
     `NotFoundError`, `ConflictError`, `UnprocessableEntityError`,
     `RateLimitError`, `InternalServerError`) plus connection/abort variants
     (`APIConnectionError`, `APIConnectionTimeoutError`, `APIUserAbortError`),
     all extending a single `APIError` base. Fern currently emits only
     `<Api>Error` and `<Api>TimeoutError`. Adding the status-family subclasses
     to the generic-sdk-error-generators is a legitimate ergonomic improvement
     (dominant convention across OpenAI / Stripe / Stainless SDKs) and would
     lift symbol coverage on all three TS fixtures simultaneously.
   - **Resource class naming.** Stainless emits resources flat at
     `src/resources/<name>.ts` exporting class `<Name>` (e.g.
     `resources/messages.ts` → `class Messages`). Fern emits them nested at
     `api/resources/<group>/.../client/Client.ts` exporting `<Name>Client`.
     This is the largest single source of missing symbols but also the most
     invasive to align — every resource method moves too.

3. File-coverage's IGNORE_PATTERNS strips license/readme/changelog/lockfiles/
   tsconfig/package.json, so adding those to the emitted dist does not move
   the file metric. They still matter for installability (the gate for the
   tier-1 behavioral metric).

4. The lumaai fern-generate failure persists from earlier rounds. The eval
   log captures no stderr (the worker run buffer is empty); diagnosing this
   requires running the CLI directly against `baselines/lumaai/fern-input/`
   and inspecting the OpenAPI parse path. Out of scope for a 30-minute round.

## Change in this commit

Extend the previous round's `package.json` fix (`copyDistTo`) by also staging
`LICENSE` and `CHANGELOG.md` into the zipped dist tree. Both are conventional
package-root artifacts that npm-aware tooling (and the eval's installable-
package gate, once a real behavioral runner lands) expects to find. The
change is additive — `cp` already swallows missing-file errors via the
existing try/catch, so packages without a LICENSE/CHANGELOG continue to
work unchanged.
