### Why I did not ship a generator code change this round

I burned the first ~10 minutes (a) discovering that prior rounds' "infra is broken"
claims are no longer true post-`20d3b8112e6`, and (b) running a baseline pair to
see what the eval actually reports. The second 10 minutes went to reading the four
non-behavioral metric implementations and the relevant generator emission points
(`generators/typescript/sdk/generator/src/SdkGenerator.ts` around line 2168;
`generators/typescript/sdk/generator/src/contexts/...`) to figure out where a
backwards-compatible change could land.

Every concrete lever I evaluated requires more than the remaining 5–7 minutes to
ship safely (compile + the legacy TS generator's snapshot tests + commit + hook):

1. **Behavioral unlock (largest lever).** Stainless honcho expects a
   `package.json` with `name: "@honcho-ai/core"` in `fern-output/typescript/`.
   Fern's TS SDK does not currently emit one in `local-file-system` mode. The
   generator does emit packaging metadata when `publishing` to an npm registry,
   so the change is plumbing `output.location: local-file-system` to also emit a
   `package.json` (configurable name from generators.yml or IR). This is gated
   work but compiles + ETE + snapshot tests are not 5-minute work in this repo.
2. **Symbol coverage.** Stainless's missing symbols include `APIError`,
   `APIClient`, `APIPromise`, `AbstractPage`, `HonchoError`. Fern emits
   `HonchoApiError`, `HonchoApiClient`, etc. The naming gap is the bottleneck,
   not the absence of behavior. Either (a) emit additional `export { … as APIError }`
   aliases (purely additive, safe) or (b) flip the default error class naming
   (breaking, needs a generator migration under
   `packages/generator-migrations/src/generators/typescript/migrations/<next>.0.0.ts`).
   Option (a) is the right round-2 target.
3. **File / structural coverage.** Stainless puts source under `src/`. Fern's
   TS SDK in local-file-system mode emits flat top-level files. The structural
   metric strips top-level `src/` before comparing, so emitting a `src/` wrapper
   in local mode is purely cosmetic against the metric — what actually matters
   is reshaping `api/resources/<name>/client/…` into the stainless layout
   `resources/<name>/…`. That is a real generator-layout change and is not safe
   in the remaining time.
4. **Translator (lumaai)** still emits invalid Fern definitions
   (`GenerationRequest is already declared in generations.yml`,
   `Type AudioGenerationRequest is not defined`). That blocker lives inside the
   eval submodule (out of worker scope; do not edit).

## Recommended round-2 lever (cheapest real lift)

Add an additive re-export of the SDK error class under the canonical stainless
name. In the TS SDK, the top-level `exports.ts` / `index.ts` emission lives in
`generators/typescript/sdk/generator/src/SdkGenerator.ts:2168-2209`. Adding
`export { <ClientName>Error as APIError } from "./errors/index.js"` is purely
additive (no migration needed), measurably lifts symbol-coverage by 1 / N for
every fixture with a typed error class, and ships in a single snapshot-test
update under `generators/typescript/sdk/generator/src/__test__/`.
