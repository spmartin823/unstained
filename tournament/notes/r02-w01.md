# Round 2, Worker 01 — Scoring Environment Issues

## Worktree state inherited from orchestrator

The orchestrator's `git worktree add` does not initialize git submodules in the
new worktree. `stainless-equivalency-eval/` is therefore empty in the worker's
worktree until the worker manually runs:

```
git submodule update --init stainless-equivalency-eval
```

Even after init, the worktree is missing:
- `node_modules/` (no `pnpm install` was run in the worktree)
- `packages/cli/cli/dist/prod/cli.cjs` (no `pnpm fern:build`)
- `stainless-equivalency-eval/baselines/` (no `pnpm --filter stainless-equivalency-eval run bootstrap`)

## Implication for scoring

`automation/tournament/src/scoring.ts` invokes:

```
pnpm --filter stainless-equivalency-eval exec tsx src/cli.ts run <fixture> <language>
```

with `cwd: job.worktree`. In a fresh worktree this will:

1. Either fail at the pnpm-filter step (no node_modules / no package on disk),
   producing exit code != 0, which scoring.ts converts to a null scorecard for
   the pair. Alternatively, the eval CLI loads but immediately errors because
   `baselines/<fixture>/` is missing.
2. Even if (1) is bypassed, `invoke-fern` requires
   `packages/cli/cli/dist/prod/cli.cjs` to exist (built artifact, not checked
   into git), so generation cannot proceed.

Result: every scored SHA in round 2 will get `{ete: "pass", pairs: {}}` with
`t1 = t2 = 0`. Tiebreak falls through to **tier 3 (earliest scoredAt)**.

## Suggested orchestrator fixes (for the human in the morning)

Add a one-time per-worktree setup before workers spawn (in
`round.seedWorkerWorktree` or a new step), or before scoring in
`scoring.scoreInner`:

```bash
git -C <worktree> submodule update --init stainless-equivalency-eval
pnpm -C <worktree> install --frozen-lockfile
pnpm -C <worktree> fern:build
pnpm -C <worktree> --filter stainless-equivalency-eval run bootstrap
```

Bootstrap is slow (clones 9 repos: lumaai, papr, honcho × {input, ts, py}); it
should be done once at orchestrator preflight against a shared eval cache and
hard-linked into worker worktrees. Otherwise every scoring cycle pays the same
clone cost.

A simpler interim fix: have the orchestrator run scoring with
`cwd: paths.parentRepoRoot` (the unstained checkout, which has the submodule
initialized and node_modules populated) and pass `--worktree` to the eval CLI
to override `fern-input/fern-output` paths into the worker's worktree. The eval
runner would need a small extension to support that.
