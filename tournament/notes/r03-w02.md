# R03-W02 Notes

## Outcome

Round 3 is unwinnable on metrics for the same reasons w02 documented in r02. All scored SHAs so far return `t1=0/t2=0` because eval pair scoring fails with all-null metrics.

## Root cause chain (verified in this round)

1. **Worktree is missing the eval submodule.** When the orchestrator creates `/Users/SeamusMartin1/tournament-worktrees/r03-w02`, the `stainless-equivalency-eval` gitlink is present but the submodule contents are not checked out. `pnpm-workspace.yaml` lists `stainless-equivalency-eval`, so without the submodule the workspace filter resolves no package and scoring fails with `No projects matched the filters in <worktree>`. Fix: run `git submodule update --init --recursive` after `worktreeAdd` in `automation/tournament/src/round.ts`. I did this manually here; the orchestrator should do it for every worker.

2. **`pnpm install` did not produce an `.install.log`.** `installDependencies` in `round.ts` writes `${logPath}.install.log` best-effort, but no log exists at `/Users/SeamusMartin1/unstained/tournament/logs/r03-w02.log.install.log` and there was no `node_modules/` when I arrived. I ran `pnpm install` manually with Node 24 and it succeeded in 21s. Either install was skipped or the orchestrator's spawn process didn't await the install before launching the worker. Worth instrumenting/confirming.

3. **Prebuilt Fern CLI is required.** `stainless-equivalency-eval/src/runner/invoke-fern.ts` shells out to `node packages/cli/cli/dist/prod/cli.cjs generate --local --group <lang>`. If that binary is missing, `invokeFern` returns `{ ok: false, error: "Fern CLI not built at ..." }` and every metric is null. The orchestrator never runs `pnpm fern:build`.

4. **`pnpm fern:build` is broken in this repo state.** The build chain depends on `^compile`, and `@fern-api/ir-migrations#compile` (which runs `tsc`) fails because `@fern-fern/ir-v66-sdk@0.0.1` was published as an empty package — only `package.json` exists in `node_modules/.pnpm/@fern-fern+ir-v66-sdk@0.0.1/node_modules/@fern-fern/ir-v66-sdk/`. No `dist/`, no `.d.ts`, no `.js`. `npm view @fern-fern/ir-v66-sdk versions` shows `["0.0.1"]` — that's the only published version. Confirmed `@fern-fern/ir-v65-sdk@0.0.5` is in the same state. These external SDKs in the `@fern-fern` org appear to be broken at the registry level.

## Why an ambient shim isn't enough

r02-w02's note proposed a `declare module` ambient shim. That unblocks source files (`packages/cli/generation/ir-migrations/src/migrations/v66-to-v65/migrateFromV66ToV65.ts`) but the migration uses dozens of named types (`IrVersions.V66.Name`, `NameOrString`, `Type`, `TypeDeclaration`, `TypeReference`, `ContainerType`, `Literal`, `NamedTypeDefault`, `V2SchemaExamples`, `FernFilepath`, `DeclaredTypeName`, `ObjectProperty`, `PropertyPathItem`, `ProtobufType`, `Source`, `ResolvedTypeReference`, `ShapeType`, `UndiscriminatedUnionMember`, `EnumValue`, `SingleUnionType`, `SingleUnionTypeProperties`, `ExamplePrimitive`, `ExampleType`, `ExampleTypeReference`, …). A blanket `any` shim works for typecheck but the runtime still needs `dist/cjs/index.js` to exist — the bundler can't pull it from thin air, and `migrateFromV66ToV65.ts` does call `V66.Type.object(...)`, `V66.ErrorDiscriminationStrategy.statusCode()`, etc. at runtime.

## Recommended fixes (ordered by leverage)

1. **Republish `@fern-fern/ir-v66-sdk` with proper dist artifacts.** This is the canonical fix. The publish pipeline that produced 0.0.1 is broken; whoever owns the IR SDK pipeline (Fern infra) needs to fix it and ship 0.0.2 or republish-as-0.0.2.
2. **Pre-bundle a working prod CLI into `main`.** Until the SDK is fixed, commit `packages/cli/cli/dist/prod/cli.cjs` (or stage it via a build that uses a vendored copy of v66-sdk) so every worker starts with a usable Fern binary. The eval just needs the binary to exist.
3. **Fix the orchestrator to init submodules + run `pnpm install` + `pnpm fern:build` once per round.** Cache the result; symlink/hardlink `dist/prod/cli.cjs` into each worker's worktree. This isolates workers from the broken upstream SDK as long as someone can get one prod build to land.
4. **Vendor v66-sdk locally.** Fork the source (the schema is in `packages/ir-sdk/fern/apis/ir-types-latest/`), generate it locally with the TS generator, publish to a local registry, and override in the workspace. Heavyweight but unblocks scoring without waiting on Fern infra.

## CORRECTIONS (post-build)

`pnpm fern:build` actually SUCCEEDED in this worktree (Node 24, 3m14s, 95/95 tasks). The v66-sdk concern from r02-w02 may have been transient. The prod CLI binary was produced cleanly.

`pnpm --filter stainless-equivalency-eval bootstrap` also succeeded for all 3 fixtures.

The REAL blocker for non-null metrics: the eval's translate step produces Fern definitions the CLI can't parse:
- fern.config.json placed next to (not inside) fern/ dir. `getFernDirectory` does findUp("fern", {type:"directory"}) then checks `<that>/fern.config.json`. Workaround: cp into fern/ dir.
- lumaai-typescript: 7 schema errors (`Type ReframeImageRequest is not defined`, etc.; `GenerationRequest is already declared in generations.yml`).
- honcho-typescript: `Unrecognized key(s) in object: 'packageName'` on generators.yml.

Root cause is now: eval translation gaps + schema drift between Stainless config and the current Fern CLI. The eval directory is off-limits to workers, so the human must patch `stainless-equivalency-eval/src/translate/`.

Recommended orchestrator changes (final):
- Pre-build + pre-bootstrap once per round, share via symlink. Workers should not each pay 5min.
- Ensure Node 24 is on the worker's PATH (the global `node` is 20.14, which pnpm 11 rejects).
- Init submodules in worktreeAdd.
