## Root cause #1 — fixed in this commit (`1e554fcde2b`)

The eval's translate step writes:

```
baselines/<fixture>/fern-input/
├── fern.config.json
└── fern/
    ├── generators.yml
    └── openapi/openapi.json
```

…with `fern.config.json` placed **next to** the `fern/` directory, not inside
it. (Stainless layout convention vs. Fern's.) The eval is off-limits, so the
fix had to be on the CLI side.

`getFernDirectory` was rejecting that layout because it required
`fern/fern.config.json`. I made two changes in
`packages/cli/configuration-loader/`:

1. `getFernDirectory.ts` — accept the fern/ dir when `fern.config.json` exists
   either inside it OR next to it.
2. `fern-config-json/loadProjectConfig.ts` — fall back to the parent directory
   when `fern.config.json` isn't found inside the resolved fern dir.

Both are purely additive — they don't change behavior for projects that follow
the canonical layout. Pre-commit hook (Node 24) passed cleanly.

Changelog: `packages/cli/cli/changes/unreleased/accept-fern-config-next-to-fern-dir.yml`
(type: fix).

**Note on allowlist:** these edits are under `packages/cli/**` which is
outside the auto-merge allowlist. The merge will require human review. I chose
this path because (a) the bug is unambiguous and tiny, (b) no allowlist path
can fix CLI config discovery, and (c) `r03-w02` already documented this is
THE blocker. Worth a PR review.

## Root cause #2 — next blocker (not fixed)

With the CLI fix in place, `fern generate --local --group <lang>` runs
end-to-end but each fixture hits its own generator-side schema rejection.
Confirmed for two:

- **honcho-python** (`fernapi/fern-python-sdk:4.21.0`): generator rejects
  `project_name: honcho-core` in `generators.yml` config →
  `pydantic_core._pydantic_core.ValidationError: 1 validation error for SDKCustomConfig — project_name — Extra inputs are not permitted`.
  Source is `generators/python/src/fern_python/generators/sdk/custom_config.py:112`.
- **honcho-typescript** (`fernapi/fern-typescript-sdk:3.49.0`): also fails
  generate after the CLI dir-lookup change (8s duration, vs. 1s pre-fix), but
  I did not capture the specific stderr in time.

Fixing these is the next leverage point. **Path-allowlist note:** the Python
generator source IS in `generators/python/`, but the eval invokes the
**published Docker image** (`fernapi/fern-python-sdk:4.21.0`), not your local
source. Editing `custom_config.py` won't take effect until a new image is
published. Same for TypeScript (`fernapi/fern-typescript-sdk:3.49.0`).

So unblocking behavioral metrics realistically requires one of:
1. Publish a new generator Docker image with looser custom-config schema
   (out of band of any worker's session).
2. Get the eval to override the generator version in `generators.yml` to point
   at a Fern-version-controlled image — eval is off-limits.
3. Add a CLI-side filter that strips/normalizes generator config fields before
   shipping to the Docker container (would need to be carefully gated; unclear
   if eval's intent survives).

(1) is the cleanest, but the publishing pipeline is upstream of this repo.

## Worktree setup state

The orchestrator still does NOT do these for the worker; I did them manually:

1. **Submodule init.** `git submodule update --init --recursive` after worktree
   create. Both `stainless-equivalency-eval` and `marketing-site` came in as
   gitlinks only.
2. **Node 24 on PATH.** Default `node` here is 20.14, which pnpm 11 rejects.
   I used `export PATH=$HOME/.nvm/versions/node/v24.15.0/bin:$PATH`.
3. **Bootstrap eval baselines.** `pnpm --filter stainless-equivalency-eval bootstrap`
   succeeded for all 3 fixtures × 2 langs in ~1 min after submodule init. The
   r03-w02 note about fern.config.json placement is now structural in the
   eval, not transient.
4. **Build prod CLI.** `pnpm fern:build` (~4 min cold, ~1.5 min warm). The
   v66-sdk concern from r02-w02 was indeed transient; the build is clean now.

Recommended orchestrator changes (still): do all of the above once per round
and share artifacts across workers via symlink or hardlink so each worker
doesn't burn 5+ min on bootstrap/build before any real work.
